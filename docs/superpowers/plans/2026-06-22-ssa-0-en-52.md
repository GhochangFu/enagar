## Exit criteria (sprint gate)

SSA-0 is **done** when all of the following are true:

1. **ADR accepted** — `ADR-0016` status is `Accepted` with date and decision-makers filled.
2. **Sahayak boundary documented** — explicit list of what SSA reuses vs does not reuse from `chatbot` module (no RAG v1, no auto-publish, staff audit).
3. **LLM env documented** — `LLM_PROVIDER` reuse, optional `SETUP_ASSISTANT_MAX_TOKENS_PER_SESSION`, DPA policy (`SETUP_ASSISTANT_SKIP_DPA_DEV` for local dev).
4. **Workflow replace/merge documented** — user-instruction-driven; maps to `replaceWorkflowDraft` / `mergeWorkflowDraft` tools in SSA-3.
5. **Archetype catalog complete** — all five v1 archetypes mapped to real seed functions with file paths verified against repo.
6. **Spec cross-links** — ADR references spec §2 decisions; spec references ADR-0016.
7. **Review sign-off** — spec compliance review ✅ and ADR quality review ✅ (see Testing & review below).

## Task 0.1: Write ADR-0016

**Files:**

- Create: `docs/ADRs/ADR-0016-service-setup-assistant.md`
- Modify: `docs/superpowers/specs/2026-06-22-service-setup-assistant-design.md` (add ADR link in Related)

- [ ] **Step 1: Header + Context**

Record problem (Service Designer complexity), personas (`tenant_admin`, `state_admin`), and relationship to ADR-0008 (LLM adapter), ADR-0004 (workflow), EN-4 (global forms).

- [ ] **Step 2: Decision — module boundaries vs Sahayak**

| Topic         | Service Setup Assistant                     | Sahayak citizen chatbot       |
| ------------- | ------------------------------------------- | ----------------------------- |
| Users         | Staff (`staff_subject_id`)                  | Citizens                      |
| RAG           | None in v1                                  | KB retrieval                  |
| Auto-publish  | Never                                       | N/A                           |
| Audit table   | `service_setup_audit_logs`                  | `chatbot_audit_logs`          |
| Consent / DPA | Reuse tenant `dpa_signed`; dev skip via env | `assertDpaAllowsProviderCall` |
| Guardrails    | Admin context (no publish, no cross-tenant) | Citizen PII redaction         |

- [ ] **Step 3: Decision — LLM configuration**

- Reuse `ChatbotLlmService` + `ILLMProvider` (ADR-0008 precedence unchanged).
- Optional per-deployment cap: `SETUP_ASSISTANT_MAX_TOKENS_PER_SESSION` (default unset = no hard cap in dev).
- Local dev bypass: `SETUP_ASSISTANT_SKIP_DPA_DEV=true` when `NODE_ENV !== 'production'` (parallel to `CHATBOT_DPA_SKIP_DEV`).

- [ ] **Step 4: Decision — workflow regenerate semantics**

- **Replace** when user says "start over", "replace template", "use scrutiny template" → `replaceWorkflowDraft` or `applyWorkflowTemplate` + persist.
- **Merge** when user says "add stage", "remove payment step" → `mergeWorkflowDraft`, `addWorkflowStage`, etc.
- Model infers mode; admin can clarify in follow-up.

- [ ] **Step 5: Appendix — archetype catalog**

| Archetype          | Workflow seed function                                 | Source file                                            | Typical payment     |
| ------------------ | ------------------------------------------------------ | ------------------------------------------------------ | ------------------- |
| `linear_approval`  | `createLinearWorkflowDraft(serviceCode)`               | `packages/workflow/src/index.ts`                       | Deferred or upfront |
| `scrutiny`         | `applyHoardingScrutinyTemplate(draft)`                 | `apps/admin-tenant/lib/workflow-designer-templates.ts` | Multi-stage fees    |
| `certificate`      | `createLinearWorkflowDraft` + certificate stage effect | `packages/workflow/src/index.ts`                       | Upfront             |
| `booking`          | `applyBookingHallTemplate(draft, serviceCode)`         | `apps/admin-tenant/lib/workflow-designer-templates.ts` | Upfront + assets    |
| `municipal_ladder` | `applyMunicipalLadderTemplate(draft)`                  | `apps/admin-tenant/lib/workflow-designer-templates.ts` | Designation stages  |

- [ ] **Step 6: Alternatives, consequences, compliance, references**

Follow `docs/ADRs/template.md` structure.

## Code review gates

SSA-0 uses **document review** instead of PR code review. Two-stage review per subagent-driven-development:

### Gate R1 — Spec compliance review

Reviewer checks:

- [ ] All exit criteria D1–D4 delivered
- [ ] No product code accidentally added
- [ ] Archetype table matches repo (T1 pass)
- [ ] Spec §2 fully covered (T2 pass)
- [ ] Contradictions with ADR-0008, ADR-0004, ADR-0014 none found

**Fail action:** Revise ADR; re-run T1–T2.

### Gate R2 — ADR quality review

Reviewer checks:

- [ ] Decision is actionable for SSA-1 implementers (module path, auth pattern named)
- [ ] Alternatives table has ≥2 rejected options with reasons
- [ ] Consequences include negative costs (LLM OpEx, prompt injection risk)
- [ ] Compliance section names CI checks for SSA-6 (`pnpm test:security`)
- [ ] Prose is clear for non-author engineers

**Fail action:** Revise ADR prose; re-run R1.

### Gate R3 — Sponsor / tech lead sign-off (human)

- [ ] ADR Status set to `Accepted` after review
- [ ] EN-52 epic updated in Jira with SSA-0 complete
